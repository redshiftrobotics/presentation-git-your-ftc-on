Commits

The working directory is what you see in your editor.

The staging area is where you queue changes to commit them.

A commit is a snapshot of your repository at a particular point in time.

A picture is worth 1000 words (assumes x86)

Commit some changes

git add takes stuff in the working directory and stages it

You need to git add every time you change stuff - not just when you add new files!

git commit commits the changes in the staging area

Example

$ git status
On branch master

Initial commit

Untracked files:
 (use "git add <file>..." to include in what will be committed)
 README.md
 
nothing added to commit but untracked files present (use "git add" to track)

$ git add README.md

$ git commit -m "Initial commit"
[master (root-commit) 85d721a] Initial commit
 1 file changed, 1 insertion(+)
 create mode 100644 README.md

What just happened?

We made some changes

We staged them, with git add

We committed, with git commit

$ git status
On branch master
nothing to commit, working directory clean

$ git log

Author: Alex Jordan 
Date:		Wed Sep 23 17:39:26 2015 -0700

  Initial commit

We did it!

We used git status, because we always use git status

Then we used git log to see a log of all our commits (there's only one so far)

Making branches

A branch is a way to develop something in parallel

You can make changes and commit on a separate branch, without affecting everyone else

You already have a branch: master

git status:

$ git status
On branch master
nothing to commit, working directory clean

Let's list our branches:

$ git branch
* master

Create a new one:

$ git branch something-else

And list our branches again:

$ git branch
* master
  something-else

Finally, we can switch to our new branch:

$ git checkout something-else
Switched to branch 'something-else'

$ git status
On branch something-else
nothing to commit, working directory clean

Remotes

A remote is basically a URL + name

The URL points to some other copy of the repository

For example: GitHub

Putting them together

We're going to add a remote called origin that points to GitHub

(origin is the traditional name for your "main" remote)

Then we're going to upload our commits by pushing them to GitHub

Example

$ git push origin master

GitHub-specific stuff

If you clone someone else's repo and push, you'll be denied

Forking creates another copy of the repository, so that you can have read-write access

Protip: remotes are very useful when dealing with forks!

A Pull Request requests someone else to merge changes in your fork (use branches!)

Other useful tools

git clone will take a repository URL and download and set it up for you

git diff will show you the difference between your working directory and your staging area

git diff --cached does the same, but for the staging area and the current commit
